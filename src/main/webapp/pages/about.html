About This Project

Stream2Grid is a proof-of-concept that takes a small live feed from Twitter's status sample stream & feeds that information into various queues for processing. The information is then parsed & placed into an in-memory data grid linked to a rules engine session that identifies trending hashtags and users across the stream sample.

 

Twitter API

The Twitter API is used to collect status updates from the provided "garden hose" sample feed. Twitter4J API wrapper is used to simplify interacting with the API layer.

RabbitMQ

RabbitMQ queues are used to temporarily hold status update info by category until they can be marshalled and stored in the data grid. Given the speed of the live Twitter update stream, this step prevents latency and stalling of API queries.

Hazelcast

Hazelcast data grid is used to store status, hashtag, and user info in memory for rules trend processing. Key-Value stored are used to track users, hashtags, delete requests, and more, all coming from the live stream.

Drools Rules Engine

Expert (core engine) module is utilized to generate stateful sessions which parse trending information from the stream-store data grid. The rules base can be expanded to better earmark points of interest which could then be directly queried against the Twitter API to get more detailed results and better trend analysis.

AngularJS

AngularJS is used to serve up the responsive front-end presentation. This is my first go with the framework and I'm definitely a fan. The routing is on par with my expectations coming from BackBone, but the HTML integration is much sleeker than templating.

Amazon EC2

EC2 instances are used for hosting the RabbitMQ & Hazelcast servers. An EBS-backed medium sized Ubuntu server hosts both services. This could easily be spun up to accomodate larger stream loads by implementing more nodes and placing an EC2 load balancer in front of each cluster.

Spring Framework

Spring MVC, AMQP, & Web are in use. It's a Java project...chances are always good Spring has their hand in the stack somewhere, amirite?

Maven

Maven is used for project dependency management. Nobody likes downloading or packaging jars. That's evil. We don't like to do evil things, we just fight local repo caching instead.

SLF4J

Logging via custom annotation injecting a SLF4J logger into classes. Nothing fancy for this project such as rollover logging or multiple appenders, but the avenues are there should I choose to expand the project.

Mockito

Mockito testing framework is used for all mock unit tests and integration stubbing. If you've used one, you've used most, but I'm pretty fond of Mockito - the documentation is clean and functionality slightly more extensive than others, but static mocking remains an issue, for which I've become accustomed to EasyMock and PowerMock to overcome.
